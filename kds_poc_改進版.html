1. Executive Summary（執行摘要）

KDS 語音狀態控制是一個為多人廚房設計的語音 AI 解決方案，用語音指令替代手工觸發，解決廚房衛生、出餐效率和數據準確性三個層級的問題。

核心價值

讓廚師在製作完成的當下即時更新訂單狀態，消除手動更新的摩擦力，使 KDS 數據成為真正可用的業務決策依據。

目標客戶

15-50 座位、已使用 KDS、廚房 3 人以上、在意出餐速度與員工效率的餐廳（如松屋等類型）。

POC 計畫

以無 AI 版本開始測試，證明核心價值；後續升級 AI 版本，擴大適用場景。

2. 項目背景與現狀

📊 KDS 市場現狀

目前 KDS 系統用戶基數小，且現有系統操作流程複雜。廚師從製餐開始到出餐完成，需要經過多個步驟才能更新訂單狀態，無法實現「一鍵完成」。這種繁瑣的操作流程是導致 KDS 數據不準確的根本原因，也是市場對「更簡單、更快速的解決方案」的核心需求。

💡 語音 AI 技術機會

目前市場上還沒有成熟的餐飲廚房語音控制解決方案，但多家廠商都已經宣稱正在開發中。這意味著我們面臨一個「先發制人」的市場窗口——誰能最快推出可用的 POC，就有機會搶先佔領市場。

🚀 我們的技術優勢

✅ 已有 KDS 基礎

現有 KDS 系統可直接集成語音控制層，無需從零開始

🎓 與台大資工合作

已與台灣大學資工系合作開發 Web 線上語音點餐系統，積累了語音識別、自然語言處理的核心技術

🤖 LLM 模型整合

可利用 Large Language Model (LLM) 提升語義理解能力，使系統更智能、更易用

🔊 TTS/STT 整合準備

已確定整合方案，可快速部署文字轉語音 (TTS) 和語音轉文字 (STT) 功能

🗺️ 產品演進路線

Web/Kiosk 點餐 → 內場 KDS 語音控制 → 完整廚房工作流自動化

目前的 Web 線上語音點餐是前台場景，KDS 語音狀態控制是內場場景。兩者使用相同的核心技術棧，但應用場景不同。成功的 KDS 語音解決方案，未來可延伸至：

  • 廚房各工作站的語音協調
  • 前後場聯動的自動化流程
  • 基於語音數據的廚房效率分析

⏰ 時間和資源

  • 開發週期：1 個月內推出 POC 版本
  • 優先級：搶在競爭對手之前推出產品，建立市場先發優勢
  • 團隊配置：與台大資工系合作，確保技術可行性和創新性

3. 核心問題與機會

🔴 問題陳述

現今廚房內場流程中，師傅完成一道菜後，需要走去點擊 KDS 螢幕才能更新訂單狀態。這個動作帶來三個問題：

1️⃣
衛生問題
2️⃣
效率延遲
3️⃣
數據不準確

💎 三層價值定位

🥉 小價值：衛生 & 快速

  • 痛點：不用手碰螢幕 → 符合衛福部禁止觸碰食物相關法規
  • 易於理解：直觀的衛生價值主張
  • 局限：客戶容易理解但不夠強，缺乏商業驅動力

🥈 中價值：即時更新 → 前場體驗改善

  • 痛點：語音讓更新發生在「製作完成的當下」（師傅的自然節點）
  • 對比：手動更新發生在「有空的時候」，尖峰期落差 3-5 分鐘
  • 影響：前場等餐體驗完全不同，直接影響翻桌率和評分
  • 商業意義：老闆在意出餐速度，這直接回應其核心 KPI

🥇 大價值：數據準確性 → 真正的經營優化

  • 痛點：KDS 的數據品質完全取決於師傅更新的及時性
  • 解決：語音讓更新零摩擦 → 更新及時率提升 → 後台數據變得有意義
  • 應用：老闆能用「品項製作時間」真正優化：菜單設計、人員排班、原料備料
  • 最高價值：從經驗決策 → 數據驅動決策，這是最少人想到但最有商業價值的部分

4. 解決方案概述

方案一：無 AI 版本（基礎版・POC 首選）

工作原理

  • 師傅說出完整的、預定義的詞彙（如「雞腿飯完成」、「榻榻米炒飯出餐」）
  • 系統進行精確詞匹配，直接對應 KDS 中的訂單狀態
  • 不需要自然語言理解，邏輯簡單、穩定

✅ 優勢

  • 開發難度低、部署快速
  • 適合規模小、品項單一的餐廳
  • 零網路依賴（可本地部署）
  • 成本低

❌ 限制

  • 廚師必須使用標準用語，容易出錯
  • 菜名變化、規格差異需要額外配置
  • 無法理解上下文和自然表達

方案二：AI 版本（進階版）

工作原理

  • 師傅可用自然語言表達（「好了」、「這份雞腿飯可以上了」、「炒飯都好」）
  • LLM 模型理解語義，結合當天訂單 context 推斷具體品項
  • 系統自動對應正確的訂單狀態

✅ 優勢

  • 理解能力強，適應場景廣
  • 廚師無需記憶特定詞彙
  • 可處理菜名變化、規格差異
  • 語義辨識能力強，誤觸發率低

❌ 限制

  • 開發複雜，需要訓練語音模型
  • 可能需要網路連接
  • 成本更高
  • 需要足夠訂單 context 進行推斷

兩版本對比

維度 無 AI 版 AI 版
開發難度 低 高
語音理解 精確詞匹配 自然語言理解
適用場景 簡單、標準化 複雜、多樣化
廚師學習成本 高（需記住詞彙） 低（自然表達）
菜名變化適應 需重新配置 自動適應
部署速度 快 慢
成本 低 高
推薦客戶 POC 驗證、小規模餐廳 規模餐廳、複雜場景

5. 產品需求詳解

5.1 使用場景

核心場景：多人廚房內場、多站台、工作繁忙時段

具體情境

  • 廚師雙手被占用（拿著食物、器具）
  • 多人同時作業，無法離開工作站去點螢幕
  • 衛生要求不允許直接接觸螢幕
  • 訂單急件多、需要快速反饋
  • 內用 + 外送訂單混合，協調壓力大

5.2 目標用戶（POC 首選）

企業檔案

  • 店型：類似松屋等連鎖定食/快餐廳
  • 規模：15-50 座位
  • 廚房人數：3 人以上
  • 現有系統：已經在使用我們的 KDS
  • 訂單來源：同時有內用 + 外送

5.3 核心功能流程

語音 KDS 的核心流程分為 12 個環節，從廚師說話的那一刻起，到訂單狀態在系統中更新為止。

第一階段：語音捕捉 & 識別

1️⃣

喚醒詞檢測

  • 系統持續監聽麥克風輸入，識別喚醒詞（如「好了」、「完成」）
  • 喚醒詞確認後，啟動主動錄音模式
  • 避免日常對話被誤觸發的機制
2️⃣

語音採集

  • 廚師佩戴的無線麥克風（小蜜蜂）開始錄音
  • 捕捉廚師的完整語音命令（喚醒詞 + 品項名稱）
  • 實時發送音頻流到後端系統
3️⃣

音源識別

  • 系統識別語音來自哪支麥克風（對應的音源 channel）
  • 根據麥克風 ID 確定說話者身份和所屬工作站
  • 建立「誰說的」→「負責訂單」的對應關係

第二階段：語音轉文字 & 語義理解

4️⃣

語音轉文字 (STT - Speech-to-Text)

  • 將廚師的語音轉換成文字
  • 例如：「好了，雞腿飯」→ 文字 "好了，雞腿飯"
  • STT 應在低延遲環境下完成（< 1 秒）
5️⃣

語義理解 (AI 版本使用)

  • LLM 模型接收文字輸入，理解廚師的實際意圖
  • 例如：「炒飯都好」→ 理解為「所有炒飯製作完成」
  • 結合當天訂單 context（訂單列表、時間、狀態），推斷具體是哪些訂單
6️⃣

詞彙匹配 (無 AI 版本使用)

  • 系統與預定義的詞彙表進行精確匹配
  • 如果完全匹配，則確定對應的訂單 ID

第三階段：訂單對應 & 驗證

7️⃣

訂單查詢

  • 根據麥克風對應的廚師身份和工作站，查詢當前未完成的訂單列表
  • 縮小搜尋範圍，提升匹配準確度
8️⃣

狀態驗證

  • 確認識別出的品項名稱是否與待完成訂單匹配
  • 檢查識別信心度（confidence score）
  • 如果信心度 > 90%，自動更新；如果 < 90%，進入確認流程
9️⃣

人工確認流程

  • 系統文字轉語音 (TTS)，用語音詢問廚師：「確認完成的是雞腿飯嗎？」
  • 廚師回答「是」或「否」
  • 保障在複雜環境（多人、噪音）下的準確性

第四階段：狀態更新 & 通知

🔟

KDS 狀態更新

  • 訂單狀態從「製作中」更新為「完成」或「待取餐」
  • 同步更新 KDS 系統的顯示畫面
  • 觸發前場通知邏輯
1️⃣1️⃣

數據記錄

記錄以下信息用於分析和優化：

  • 語音命令的原始文字
  • 識別時間戳（精確到秒）
  • 廚師 ID 和工作站 ID
  • 對應的訂單 ID
  • 識別信心度和處理耗時
1️⃣2️⃣

前場通知

系統通知前場員工：「雞腿飯（訂單 #123）已完成，可出餐」

可通過多種方式通知：

  • KDS 屏幕亮起
  • 聲音提示（「某某餐點已完成」）
  • 手機 push 通知（如使用 App）

關鍵性能指標

環節 目標耗時 備註

STT → 語義理解 < 0.5s 無 AI 版本瞬間完成

6. 技術開發注意事項

6.1 喚醒詞機制設計

設計原則

  • 選擇明確、獨特的喚醒詞（如「開始」、「好了」、「完成」）
  • 喚醒詞不能與菜名衝突
  • 支持多個喚醒詞組合，增加識別穩定性

6.2 多麥克風音源隔離 ⭐ 重點

問題

多人廚房同時說話，系統需要知道「是誰說的」

解決方案

  • 每支無線麥克風（小蜜蜂）對應一個獨立音源 channel
  • 後端需要音源定位能力，區分不同麥克風的輸入
  • 根據麥克風 ID → 對應該廚師負責的訂單

6.3 菜品同名辨識

無 AI 版本

  • 廚師說出完整的訂單代號或規格名稱
  • 例如：「A12 雞腿飯完成」

AI 版本

  • LLM 根據當日訂單 context 推斷
  • 結合時間戳、訂單狀態、廚師負責區域進行消歧

6.4 其他技術考量

7.4 POC 時間軸

第 1 週：硬體部署 & 廚師培訓
  • Day 1-2：無線麥克風安裝、KDS 連接
